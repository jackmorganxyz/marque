# Marque + Flue quick start — zero to deployed signing agent

> **Audience: AI coding agents.** This file is self-contained — paste it (or point your agent at it) and it walks you from an empty directory to a **deployed** [Flue](https://flueframework.com/) agent on Cloudflare Workers that signs every outbound message and verifies every inbound message against an allowlist. Humans welcome too.

Marque lets this agent **sign** outbound messages with its own secp256k1 key and **verify** which https origin sent any inbound message — no shared secret, no registry, no chain. Identity is anchored at `https://<origin>/.well-known/marque.json`, served from the agent's own domain over TLS.

The path: **two questions for the operator → scaffold → install → keys → two files → build check → customize → deploy to Cloudflare → verify live.** Follow the steps in order; if one fails, stop and report it instead of improvising.

## Step 0 — two questions for the operator

Ask both now, together, so the rest runs without interruption:

1. **Domain** — what bare hostname will this agent live at? (`eve.example.com` — never a URL, never `localhost`, never an IP, never a port.) If they have no custom domain, Cloudflare will assign `<name>.<subdomain>.workers.dev` at Step 7 — use that as the origin, but tell the operator a custom domain is safer (subdomain-takeover risk is documented in the README's Security section).
2. **Allowlist** — which sender hostnames should this agent trust inbound messages from? (e.g. `bob.example.com,carol.example.com`). Empty is fine to start — the inbox stays closed until it's filled. Never guess, and never leave verification open to everyone.

Also collect the **model provider key** (`ANTHROPIC_API_KEY` for the default model below) — Flue needs one to run the agent at all.

## Step 1 — scaffold the Flue project

Already inside a Flue project (`flue.config.ts` plus a source directory with `agents/`)? Skip ahead — but note this guide writes to `src/`; if the project uses the `.flue/` layout, write the same files there instead (Flue reads exactly one source directory, `.flue/` first).

Otherwise (needs Node 22.19+):

```bash
mkdir my-agent && cd my-agent
npm init -y
npm install @flue/runtime valibot hono 'agents@^0.14.2'
npm install -D @flue/cli wrangler
npx flue init --target cloudflare
```

`agents` is Cloudflare's Agents SDK (Flue builds on it); `hono` is needed because Step 4 authors a custom `src/app.ts`. Then create `wrangler.jsonc` at the project root — Cloudflare requires an explicit Durable Object migration for the agent class Flue will generate from `src/agents/assistant.ts` (Step 4):

```jsonc
{
  "$schema": "./node_modules/wrangler/config-schema.json",
  "name": "my-agent",
  "compatibility_date": "2026-06-01",
  "compatibility_flags": ["nodejs_compat"],
  "migrations": [{ "tag": "v1", "new_sqlite_classes": ["FlueRegistry", "FlueAssistantAgent"] }]
}
```

`nodejs_compat` is required twice over: Flue needs it, and Marque imports `node:net`. The recent `compatibility_date` matters too — it makes Workers populate `process.env` from vars and secrets, which every file below relies on. If you rename the agent file, rename the migration class to match (`src/agents/<kebab-name>.ts` → `Flue<PascalName>Agent`), and when you add more agents or workflows later, append a **new** uniquely-tagged migration — never rewrite a deployed one.

## Step 2 — install Marque

```bash
npm i github:jackmorganxyz/marque
```

Installs from GitHub and builds automatically; imports are still `from "marque"`. (Do NOT `npm i marque` — that unscoped npm name currently belongs to an unrelated package.) All `npx marque …` commands below resolve to the locally installed CLI after this step.

## Step 3 — key + env vars

```bash
npx marque init <origin>        # e.g. npx marque init eve.example.com
```

Prints the env vars and the well-known JSON. Confirm `.dev.vars` and `.env*` are gitignored (`printf '\n.dev.vars*\n.env*\n' >> .gitignore`), then write **five** vars to `.dev.vars` — Cloudflare's local-dev secrets file, plain `NAME=value` lines, no inline comments — the three printed, plus the allowlist and provider key from Step 0:

| var | secret? | value |
|---|---|---|
| `MARQUE_PRIVATE_KEY` | **yes** | `0x…` signing key |
| `MARQUE_ADDRESS` | no | the key's `0x…` address — comma-separate several to overlap during rotation |
| `MARQUE_ORIGIN` | no | this agent's bare hostname, e.g. `eve.example.com` |
| `MARQUE_ALLOW` | no | comma-separated sender hostnames to trust, e.g. `bob.example.com,carol.example.com` |
| `ANTHROPIC_API_KEY` | **yes** | the model provider key from Step 0 |

Rules for the private key — no exceptions:
- **Never** commit it (that's why you confirmed `.dev.vars` is gitignored first).
- **Never** print it in code, logs, comments, chat output, or a client-visible binding.
- It must be a **dedicated key** generated by `marque init` — never a key also used for wallets, SIWE, or anything else.

## Step 4 — two files

### 4a — the agent + outbound signing tool: `src/agents/assistant.ts`

The filename becomes the agent name (`assistant`), and Flue generates the `FlueAssistantAgent` class the Step 1 migration already names. **No `route` export, on purpose** — the verified inbox in 4b is the only public way in; a pass-through `route` would expose an unauthenticated prompt endpoint.

```ts
import { defineAgent, defineTool } from "@flue/runtime";
import * as v from "valibot";
import { sign } from "marque";

export const description = "Signs outbound agent-to-agent messages with Marque and acts on verified inbound ones.";

const sendSignedMessage = defineTool({
  name: "send_signed_message",
  description: "Send a message to another agent, signed so they can verify it came from us.",
  input: v.object({
    url: v.pipe(v.string(), v.url(), v.description("recipient inbox endpoint, e.g. https://bob.example.com/inbox")),
    toOrigin: v.pipe(v.string(), v.description("recipient bare hostname, e.g. bob.example.com")),
    payload: v.any(),   // the message — JSON-safe values only (see Failure modes)
  }),
  async run({ input, signal }) {
    const signed = await sign(input.payload, {
      privateKey: process.env.MARQUE_PRIVATE_KEY as `0x${string}`,
      origin: process.env.MARQUE_ORIGIN!,   // us
      aud: input.toOrigin,                  // them — envelope is bound to this one recipient
    });
    const res = await fetch(input.url, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify(signed),
      signal,
    });
    return { delivered: res.ok, status: res.status };
  },
});

export default defineAgent(() => ({
  model: "anthropic/claude-sonnet-4-6",
  instructions:
    "When sending a message to another agent, always use the send_signed_message tool " +
    "(never plain fetch) so the recipient can verify it came from us. Inbound messages " +
    "arrive already verified at /inbox; the sender's identity accompanies the payload.",
  tools: [sendSignedMessage],
}));
```

### 4b — publish identity + verified inbox: `src/app.ts`

The well-known route serves the agent's **address** (never the private key). The inbox rejects forgeries before the model ever sees them: `MARQUE_ALLOW` is the allowlist from Step 0 — a valid signature from an origin you don't trust is still a stranger, so `verify` rejects it. Only then does `dispatch(...)` hand the payload to the agent, one continuing session per sender.

```ts
import { dispatch } from "@flue/runtime";
import { flue } from "@flue/runtime/routing";
import { Hono } from "hono";
import { verify } from "marque";
import assistant from "./agents/assistant.ts";

// ⚠ Per-isolate replay guard. Workers isolates don't share memory: either accept
// replay within the 300s window to this audience, or back `seen` with a shared
// store (Durable Object / Workers KV) exposing Map-like has/set/delete/iteration.
const seen = new Map<string, number>();

const app = new Hono();

app.get("/.well-known/marque.json", (c) => {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map(s => s.trim()).filter(Boolean);
  return c.json({ v: 1, keys }, 200, { "cache-control": "public, max-age=300" });
});

app.post("/inbox", async (c) => {
  const msg = await c.req.json();
  const allow = (process.env.MARQUE_ALLOW ?? "").split(",").map(s => s.trim()).filter(Boolean);
  const r = await verify(msg, { selfOrigin: process.env.MARQUE_ORIGIN!, seen, allow });
  if (!r.ok) return c.json({ error: "marque: " + r.reason }, 401);
  // Authenticated + allowlisted: deliver payload + identity to the agent's session.
  const receipt = await dispatch(assistant, {
    id: r.identity.replace(/^https:/, ""),           // one agent instance per sender
    input: { type: "marque.message", from: r.identity, payload: msg.payload },
  });
  return c.json({ ok: true, from: r.identity, ...receipt }, 202);
});

app.route("/", flue());

export default app;
```

## Step 5 — build check, then a local look

Confirm everything compiles and the generated Worker config passes Wrangler's dry run:

```bash
npx flue build --target cloudflare
npx wrangler deploy --dry-run --config dist/my_agent/wrangler.json
```

(The generated directory is the Worker name from `wrangler.jsonc`, snake_cased — `my-agent` builds to `dist/my_agent/`; confirm with `ls dist`.) Optionally eyeball the well-known locally — `npx flue dev --target cloudflare` reads `.dev.vars` and serves on port 3583:

```bash
curl http://localhost:3583/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}
```

Don't POST to the local `/inbox` yet: `verify` fetches the sender's key from the **live** `https://<origin>/.well-known/marque.json`, so inbound verification only round-trips after Step 7.

## Step 6 — ask the operator: what should this agent actually do?

The signing/verification skeleton is done. Before deploying, ask the operator what makes this agent special:

- **New tools** — more `defineTool` entries (same shape as Step 4a) for APIs it should call or actions it should take.
- **Personality / behavior** — expand the `instructions` string (long instructions can move to a markdown file imported `with { type: "markdown" }`).
- **Peers** — more origins for `MARQUE_ALLOW`.
- **Direct access** — only if the operator wants to prompt the agent over HTTP themselves, add an authenticated `route` export (`AgentRouteHandler`) to `src/agents/assistant.ts`; never a bare pass-through on an internet-facing agent.

Implement what they ask, rerun the Step 5 build check, and leave the Step 4 signing/verification code untouched.

## Step 7 — deploy to Cloudflare

Suggest deploying now. On a yes:

```bash
npx wrangler login        # opens browser auth — skip if already logged in
npx flue build --target cloudflare
npx wrangler deploy --config dist/my_agent/wrangler.json
```

Always deploy the **generated** config from `dist/`, not the project-root `wrangler.jsonc`. The first deploy prints the Worker URL. No custom domain? Your origin is `my-agent.<subdomain>.workers.dev` — update `MARQUE_ORIGIN` in `.dev.vars` to it (the key doesn't change; no need to rerun `marque init`). Custom domain? Add it to the project-root `wrangler.jsonc` and rebuild + redeploy:

```jsonc
"routes": [{ "pattern": "eve.example.com", "custom_domain": true }]
```

Mirror the five vars to the deployed Worker as secrets without printing them (`wrangler secret put` targets the Worker named in the project-root `wrangler.jsonc` and takes effect immediately, no redeploy; skip any empty value — unset and empty behave the same):

```bash
set -a; source .dev.vars; set +a    # load the five vars into the shell quietly
printf '%s' "$MARQUE_PRIVATE_KEY" | npx wrangler secret put MARQUE_PRIVATE_KEY
printf '%s' "$MARQUE_ADDRESS"     | npx wrangler secret put MARQUE_ADDRESS
printf '%s' "$MARQUE_ORIGIN"      | npx wrangler secret put MARQUE_ORIGIN
printf '%s' "$MARQUE_ALLOW"       | npx wrangler secret put MARQUE_ALLOW
printf '%s' "$ANTHROPIC_API_KEY"  | npx wrangler secret put ANTHROPIC_API_KEY
```

## Step 8 — verify the live deployment

```bash
curl https://<origin>/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}

# Full round trip against the live well-known (sign to self, verify as self):
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> | npx marque verify <origin>
# → {"ok":true,"identity":"https:<origin>","signer":"0x…"}
```

If the round trip returns `"key not published at origin"`, the crypto worked — only the well-known fetch failed (not deployed yet, wrong origin, or cached stale for up to 60s). To exercise the inbox end-to-end, temporarily add `<origin>` itself to `MARQUE_ALLOW` and POST a self-signed envelope (this dispatches a real message to the agent, so it spends one model call):

```bash
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> \
  | curl -sS -X POST https://<origin>/inbox -H 'content-type: application/json' -d @-
# → {"ok":true,"from":"https:<origin>",…}
```

Finish by reporting to the operator: the agent's origin, its published address, the current allowlist, and that peers must add this origin to *their* allowlist before they'll accept its messages.

## Optional — X handle verification

Skip freely; nothing above depends on it. This binds the agent's signing key to an X (Twitter) handle as a **second, lower-assurance factor** on top of the TLS identity, published in the same well-known file (full protocol and caveats: README, "X handle verification"). Only do this if the operator asks for it and controls the X account.

**Build the proof** (needs `MARQUE_PRIVATE_KEY`):

```bash
set -a; source .dev.vars; set +a
npx marque link-x <handle>        # prints the proof tweet + the marque.json "x" entry
```

**Publish it.** Have the operator post the printed tweet from that X account, then add two vars to `.dev.vars` — and mirror them to the Worker like the Step 7 secrets:

```
MARQUE_X_HANDLE=<handle>
MARQUE_X_PROOF=https://x.com/<handle>/status/<tweet-id>
```

**Extend the Step 4b well-known route** so it advertises the proof next to the keys (unset vars leave the file byte-identical to before):

```ts
app.get("/.well-known/marque.json", (c) => {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map(s => s.trim()).filter(Boolean);
  const { MARQUE_X_HANDLE: handle, MARQUE_X_PROOF: proof } = process.env;
  return c.json({ v: 1, keys, ...(handle && proof ? { x: { handle, proof } } : {}) },
    200, { "cache-control": "public, max-age=300" });
});
```

Rebuild + redeploy, then confirm `curl https://<origin>/.well-known/marque.json` now includes `"x":{"handle":"…","proof":"…"}`.

**Checking a peer's handle** works from any Marque receiver — after a successful `verify()`, once per peer, not per message:

```ts
import { verifyX } from "marque/x";
const x = await verifyX("bob.example.com", r.signer);
// → { ok: true, handle: "bob" }  or  { ok: false, reason: "…" }
```

Handles are mutable and recyclable: treat a verified handle as an extra attribute of the TLS identity, never as a substitute for the allowlist.

## Failure modes

| `reason` | likely cause | fix |
|---|---|---|
| `wrong audience` | sender's `aud` ≠ receiver's `MARQUE_ORIGIN` | both must be the same bare hostname |
| `expired` / `clock skew` | message older than 300s, or clocks differ >300s | resend; check server clocks |
| `replay` | same envelope delivered twice (incl. HTTP retries) | expected — sender should re-`sign`, not re-POST |
| `payload mismatch` | body mutated in transit (proxy/middleware re-serialized JSON) | pass the envelope through untouched |
| `key not published at origin` | well-known missing, stale (≤60s verifier cache), or `MARQUE_ADDRESS` doesn't match the signing key | curl the well-known; re-put the secret |
| `origin resolve failed` | origin is localhost / an IP / has a port or path — SSRF guard rejects it | use a bare public hostname |
| `sender not allowed` | sender's origin missing from `MARQUE_ALLOW` (an empty allowlist rejects everyone) | add the peer's hostname to `MARQUE_ALLOW` |
| `bad signature` / `malformed envelope` | corrupted or hand-built envelope | resend a fresh `sign()` output |
| `sign()` throws `unsafe number` / `not JSON-safe` | payload has floats ≥2^53, NaN/Infinity, Date, bigint, undefined | restrict payloads to plain JSON-safe values |

Workers-specific:

| symptom | likely cause | fix |
|---|---|---|
| `process.env.MARQUE_*` is undefined at runtime | `nodejs_compat` missing, `compatibility_date` too old, or secret never put | check the Step 1 flags; `npx wrangler secret list` |
| build fails resolving `node:net` | `nodejs_compat` missing from `compatibility_flags` | add it and rebuild |
| deploy succeeds but every route 404s | deployed the source-root `wrangler.jsonc` instead of the generated one | always `--config dist/<name>/wrangler.json` |
| deploy rejects the Durable Object class | migration missing or renamed class | Step 1: migration class must match `Flue<PascalName>Agent` |

## Hard rules

- Act **only** on `{ ok: true }` **and** an allowlist hit. Trust no envelope field before both.
- One dedicated key per agent. Never reuse, never commit, never log.
- The well-known route serves **public values only** — the address (and the optional X proof entry), never the private key. If you ever find a private key there, that's an incident, rotate immediately.
- Never add a fallback that resolves keys any other way when the https well-known fetch fails — fail closed.
- Full threat model and residual risks: README, "Security — read before shipping". Read it before changing any verification code.
