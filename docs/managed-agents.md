# Marque + Claude Managed Agents quick start — zero to a deployed signing agent

> **Audience: AI coding agents.** This file is self-contained — paste it (or point your agent at it) and it walks you from an empty directory to a **deployed** [Claude Managed Agent](https://platform.claude.com/docs/en/managed-agents/overview) that signs every outbound agent-to-agent message and verifies every inbound one against an allowlist. Humans welcome too.

Marque lets an agent **sign** outbound messages with its own secp256k1 key and **verify** which https origin sent any inbound message — no shared secret, no registry, no chain. Identity is anchored at `https://<origin>/.well-known/marque.json`, served from the agent's own domain over TLS.

**Architecture — read this first.** A managed agent runs on Anthropic's infrastructure: it has no domain of its own, and secrets must never enter its sandbox. So the Marque half lives in a small **orchestrator** you deploy — a Node service on your domain that (a) serves the well-known file, (b) verifies inbound messages and forwards them into the agent's session, and (c) holds the signing key and executes the agent's `send_signed_message` **custom tool** calls host-side. This is Anthropic's documented pattern for credential-touching tools: the container never sees the key; tool calls arrive on the session event stream and results go back the same way.

The path: **two questions for the operator → scaffold → install → keys → two files → setup once + build check → customize → deploy → verify live.** Follow the steps in order; if one fails, stop and report it instead of improvising.

## Step 0 — two questions for the operator

Ask both now, together, so the rest runs without interruption:

1. **Domain** — what bare hostname will this agent live at? (`eve.example.com` — never a URL, never `localhost`, never an IP, never a port.) If they have no custom domain, Fly.io will assign `<app>.fly.dev` at Step 7 — use that as the origin, but tell the operator a custom domain is safer (subdomain-takeover risk is documented in the README's Security section).
2. **Allowlist** — which sender hostnames should this agent trust inbound messages from? (e.g. `bob.example.com,carol.example.com`). Empty is fine to start — the inbox stays closed until it's filled. Never guess, and never leave verification open to everyone.

Also collect an **`ANTHROPIC_API_KEY`** — Managed Agents run on the Claude API (beta), so a key with Managed Agents access is required.

## Step 2 — install Marque

```bash
npm i github:jackmorganxyz/marque
```

Installs from GitHub and builds automatically; imports are still `from "marque"`. (Do NOT `npm i marque` — that unscoped npm name currently belongs to an unrelated package.) All `npx marque …` commands below resolve to the locally installed CLI after this step.

## Step 3 — key + env vars

```bash
npx marque init <origin>        # e.g. npx marque init eve.example.com
```

Prints the env vars and the well-known JSON. Confirm `.env*` is gitignored (`printf '\n.env*\n' >> .gitignore`), then write **five** vars to `.env` (plain `NAME=value` lines, no inline comments) — the three printed, plus the allowlist and API key from Step 0. Step 4's setup script appends two more (the agent and environment IDs).

| var | secret? | value |
|---|---|---|
| `MARQUE_PRIVATE_KEY` | **yes** | `0x…` signing key |
| `MARQUE_ADDRESS` | no | the key's `0x…` address — comma-separate several to overlap during rotation |
| `MARQUE_ORIGIN` | no | this agent's bare hostname, e.g. `eve.example.com` |
| `MARQUE_ALLOW` | no | comma-separated sender hostnames to trust, e.g. `bob.example.com,carol.example.com` |
| `ANTHROPIC_API_KEY` | **yes** | the Claude API key from Step 0 |

Rules for the private key — no exceptions:
- **Never** commit it (that's why you confirmed `.env*` is gitignored first).
- **Never** put it in the agent's system prompt, a user message, or anything else that reaches the Anthropic API — session history is durably stored and readable for the life of the session. The key lives **only** in the orchestrator's environment.
- It must be a **dedicated key** generated by `marque init` — never a key also used for wallets, SIWE, or anything else.

## Step 4 — two files

### 4a — one-time setup: `setup.ts`

Creates the environment and the agent — the agent declares the `send_signed_message` **custom tool** (schema only; your orchestrator executes it) and carries the instructions that make the model use it. **Run this once and store the printed IDs** — never call `agents.create` per run; to change the agent later, update it in place (`client.beta.agents.update`), which creates a new version.

```ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

const environment = await client.beta.environments.create({
  name: "marque-agent",
  config: { type: "cloud", networking: { type: "unrestricted" } },
});

const agent = await client.beta.agents.create({
  name: "Marque Signing Agent",
  model: "claude-opus-4-8",
  system:
    "When sending a message to another agent, always use the send_signed_message tool " +
    "(never bash, curl, or web_fetch) so the recipient can verify it came from us. " +
    "Inbound messages arrive already verified; the sender's proven identity accompanies the payload.",
  tools: [
    { type: "agent_toolset_20260401" },
    {
      type: "custom",
      name: "send_signed_message",
      description:
        "Send a message to another agent, signed with this agent's Marque key so the recipient can verify it came from us.",
      input_schema: {
        type: "object",
        properties: {
          url: { type: "string", description: "recipient inbox endpoint, e.g. https://bob.example.com/inbox" },
          toOrigin: { type: "string", description: "recipient bare hostname, e.g. bob.example.com" },
          payload: { description: "the message — JSON-safe values only" },
        },
        required: ["url", "toOrigin", "payload"],
      },
    },
  ],
});

console.log("Append these to .env (and mirror them in Step 7):");
console.log(`ANTHROPIC_ENVIRONMENT_ID=${environment.id}`);
console.log(`ANTHROPIC_AGENT_ID=${agent.id}`);
```

### 4b — the orchestrator: `server.ts`

Serves the well-known and the verified inbox on your domain, keeps one continuing session with the managed agent, and executes its signing-tool calls host-side. The event loop follows Anthropic's client patterns: open the stream, replay history with dedupe on every (re)connect, and never let a pending `custom_tool_use` go unanswered.

```ts
import { serve } from "@hono/node-server";
import Anthropic from "@anthropic-ai/sdk";
import { Hono } from "hono";
import { sign, verify } from "marque";

const client = new Anthropic();
const AGENT_ID = process.env.ANTHROPIC_AGENT_ID!;
const ENVIRONMENT_ID = process.env.ANTHROPIC_ENVIRONMENT_ID!;

// ⚠ Per-process replay guard. Fine for one orchestrator instance; run several
// and you must back `seen` with a shared store exposing Map-like has/set/delete/iteration.
const seen = new Map<string, number>();

// One continuing session for the agent. Set ANTHROPIC_SESSION_ID to resume an
// existing session across orchestrator restarts; otherwise one is created at boot.
let sessionId = process.env.ANTHROPIC_SESSION_ID ?? "";

async function ensureSession(): Promise<string> {
  if (sessionId) return sessionId;
  const session = await client.beta.sessions.create({
    agent: AGENT_ID,
    environment_id: ENVIRONMENT_ID,
    title: "Marque agent",
  });
  sessionId = session.id;
  console.log(`session ${session.id}`);
  console.log(`watch live: https://platform.claude.com/workspaces/default/sessions/${session.id}`);
  return sessionId;
}

// The agent asked to send a signed message: sign HOST-SIDE and deliver the result back.
async function handleToolCall(
  sid: string,
  event: Anthropic.Beta.Sessions.BetaManagedAgentsAgentCustomToolUseEvent,
): Promise<void> {
  const { url, toOrigin, payload } = event.input as { url: string; toOrigin: string; payload: unknown };
  let text: string;
  try {
    const signed = await sign(payload, {
      privateKey: process.env.MARQUE_PRIVATE_KEY as `0x${string}`,
      origin: process.env.MARQUE_ORIGIN!,   // us
      aud: toOrigin,                        // them — envelope is bound to this one recipient
    });
    const res = await fetch(url, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify(signed),
    });
    text = JSON.stringify({ delivered: res.ok, status: res.status });
  } catch (err) {
    text = JSON.stringify({ delivered: false, error: String(err) });
  }
  await client.beta.sessions.events.send(sid, {
    events: [{ type: "user.custom_tool_result", custom_tool_use_id: event.id, content: [{ type: "text", text }] }],
  });
}

// Event pump: stream + history-replay with dedupe on every (re)connect, forever.
async function pumpEvents(sid: string): Promise<void> {
  const seenEvents = new Set<string>();
  const handle = async (event: Anthropic.Beta.Sessions.BetaManagedAgentsStreamSessionEvents) => {
    if ("id" in event && event.id) {
      if (seenEvents.has(event.id)) return;
      seenEvents.add(event.id);
    }
    if (event.type === "agent.custom_tool_use" && event.name === "send_signed_message") {
      await handleToolCall(sid, event);
    }
  };
  while (true) {
    try {
      const stream = await client.beta.sessions.events.stream(sid);   // stream first…
      for await (const event of client.beta.sessions.events.list(sid)) await handle(event); // …then history
      for await (const event of stream) {
        await handle(event);
        if (event.type === "session.status_terminated") {
          console.error("session terminated — unset ANTHROPIC_SESSION_ID and restart for a fresh session");
          return;
        }
      }
    } catch (err) {
      console.error("event stream dropped; reconnecting:", err);
      await new Promise((r) => setTimeout(r, 2000));
    }
  }
}

const app = new Hono();

app.get("/.well-known/marque.json", (c) => {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map((s) => s.trim()).filter(Boolean);
  return c.json({ v: 1, keys }, 200, { "cache-control": "public, max-age=300" });
});

// Inbox: rejects forgeries before the agent ever sees them. MARQUE_ALLOW is the
// allowlist from Step 0 — a valid signature from an origin you don't trust is
// still a stranger, so verify rejects it.
app.post("/inbox", async (c) => {
  const msg = await c.req.json();
  const allow = (process.env.MARQUE_ALLOW ?? "").split(",").map((s) => s.trim()).filter(Boolean);
  const r = await verify(msg, { selfOrigin: process.env.MARQUE_ORIGIN!, seen, allow });
  if (!r.ok) return c.json({ error: "marque: " + r.reason }, 401);
  // Authenticated + allowlisted: deliver payload + proven identity into the session.
  await client.beta.sessions.events.send(await ensureSession(), {
    events: [{
      type: "user.message",
      content: [{ type: "text", text: `Verified message from ${r.identity}:\n${JSON.stringify(msg.payload)}` }],
    }],
  });
  return c.json({ ok: true, from: r.identity }, 202);
});

const sid = await ensureSession();
void pumpEvents(sid);
serve({ fetch: app.fetch, port: Number(process.env.PORT ?? 3000) });
console.log(`orchestrator listening on :${process.env.PORT ?? 3000}`);
```

## Step 5 — run setup once, then the build check

```bash
node --env-file=.env setup.ts     # ONCE — append the two printed IDs to .env
npx tsc --noEmit                  # type check both files
```

Optionally boot locally (`node --env-file=.env server.ts`) and eyeball the well-known:

```bash
curl http://localhost:3000/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}
```

Don't POST to the local `/inbox` yet: `verify` fetches the sender's key from the **live** `https://<origin>/.well-known/marque.json`, so inbound verification only round-trips after Step 7. (Note the local boot already creates a real session; reuse it later by putting the printed session id in `ANTHROPIC_SESSION_ID`, or ignore it — sessions are disposable.)

## Step 6 — ask the operator: what should this agent actually do?

The signing/verification skeleton is done. Before deploying, ask the operator what makes this agent special:

- **Behavior** — expand the `system` prompt via `client.beta.agents.update(AGENT_ID, {...})` (each update creates a new agent version; never create a second agent for the same purpose).
- **More capabilities** — the full prebuilt toolset (`bash`, file ops, `web_search`, …) is already on; add MCP servers or more custom tools on the agent, handling any new custom tools in `server.ts` next to `send_signed_message`.
- **Peers** — more origins for `MARQUE_ALLOW`.

Rerun the Step 5 type check, and leave the Step 4 signing/verification code untouched.

## Step 7 — deploy the orchestrator

The orchestrator is a single long-running Node process — deploy it anywhere that gives your domain TLS and keeps the process alive. With [Fly.io](https://fly.io):

```bash
fly launch --no-deploy      # accept the Node defaults; pick the app name
fly secrets import < .env   # mirrors all seven vars (incl. the two IDs) without printing them
fly deploy
```

No custom domain? Your origin is `<app>.fly.dev` — update `MARQUE_ORIGIN` in `.env` to it, re-run `fly secrets import < .env`, and redeploy (the key doesn't change; no need to rerun `marque init`). Custom domain? `fly certs add <domain>` and point DNS at the app, keeping `MARQUE_ORIGIN` equal to that hostname.

Keep exactly **one** running instance (`fly scale count 1`) — the replay guard and the session event pump are per-process.

## Step 8 — verify the live deployment

```bash
curl https://<origin>/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}

# Full round trip against the live well-known (sign to self, verify as self):
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> | npx marque verify <origin>
# → {"ok":true,"identity":"https:<origin>","signer":"0x…"}
```

If the round trip returns `"key not published at origin"`, the crypto worked — only the well-known fetch failed (not deployed yet, wrong origin, or cached stale for up to 60s). To exercise the whole loop, temporarily add `<origin>` itself to `MARQUE_ALLOW` and POST a self-signed envelope — this sends a real message into the managed session (one model turn), which you can watch live at the Console URL the orchestrator logged:

```bash
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> \
  | curl -sS -X POST https://<origin>/inbox -H 'content-type: application/json' -d @-
# → {"ok":true,"from":"https:<origin>"}
```

Finish by reporting to the operator: the agent's origin, its published address, the current allowlist, the agent/environment/session IDs, and that peers must add this origin to *their* allowlist before they'll accept its messages.

## Failure modes

| `reason` | likely cause | fix |
|---|---|---|
| `wrong audience` | sender's `aud` ≠ receiver's `MARQUE_ORIGIN` | both must be the same bare hostname |
| `expired` / `clock skew` | message older than 300s, or clocks differ >300s | resend; check server clocks |
| `replay` | same envelope delivered twice (incl. HTTP retries) | expected — sender should re-`sign`, not re-POST |
| `payload mismatch` | body mutated in transit (proxy re-serialized JSON) | pass the envelope through untouched |
| `key not published at origin` | well-known missing, stale (≤60s verifier cache), or `MARQUE_ADDRESS` doesn't match the signing key | curl the well-known; fix secrets; redeploy |
| `origin resolve failed` | origin is localhost / an IP / has a port or path — SSRF guard rejects it | use a bare public hostname |
| `sender not allowed` | sender's origin missing from `MARQUE_ALLOW` (an empty allowlist rejects everyone) | add the peer's hostname to the secrets |
| `bad signature` / `malformed envelope` | corrupted or hand-built envelope | resend a fresh `sign()` output |
| `sign()` throws `unsafe number` / `not JSON-safe` | payload has floats ≥2^53, NaN/Infinity, Date, bigint, undefined | restrict payloads to plain JSON-safe values |

Managed-Agents-specific:

| symptom | likely cause | fix |
|---|---|---|
| `401 authentication_error` from the SDK | `ANTHROPIC_API_KEY` missing/invalid in the deployed secrets | re-import secrets; check the key has Managed Agents beta access |
| agent calls the tool but nothing sends | orchestrator event pump not running, or stream died without reconnect | check orchestrator logs; the pump reconnects every 2s — a pending `custom_tool_use` resolves on reconnect via the history replay |
| session stuck `idle` waiting | a `custom_tool_use` was never answered (orchestrator was down) | restart the orchestrator — history replay picks up the pending call |
| new agents piling up in the Console | `setup.ts` being re-run per deploy | run it once; to change the agent, `agents.update` (new version), never a new create |
| agent puts secrets in messages | key or API key leaked into prompts | never send secrets into the session; they persist in event history — rotate immediately |

## Hard rules

- Act **only** on `{ ok: true }` **and** an allowlist hit. Trust no envelope field before both.
- One dedicated key per agent. Never reuse, never commit, never log — and **never inside the session**: not in the system prompt, not in a message, not in a tool result. The sandbox must never see it.
- The well-known route serves the **address only** — if you ever find a private key there, that's an incident, rotate immediately.
- Never add a fallback that resolves keys any other way when the https well-known fetch fails — fail closed.
- Agent once, session per lifetime, `agents.update` for changes — never `agents.create` in the request path.
- Full threat model and residual risks: README, "Security — read before shipping". Read it before changing any verification code.
