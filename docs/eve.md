# Marque + eve quick start — zero to deployed signing agent

> **Audience: AI coding agents.** This file is self-contained — paste it (or point your agent at it) and it walks you from an empty directory to a **deployed** [eve](https://vercel.com/eve) agent that signs every outbound message and verifies every inbound message against an allowlist. Humans welcome too.

Marque lets this agent **sign** outbound messages with its own secp256k1 key and **verify** which https origin sent any inbound message — no shared secret, no registry, no chain. Identity is anchored at `https://<origin>/.well-known/marque.json`, served from the agent's own domain over TLS.

The path: **two questions for the operator → scaffold → install → keys → three files → one line of instructions → customize → deploy to Vercel → verify live.** Follow the steps in order; if one fails, stop and report it instead of improvising.

## Step 0 — two questions for the operator

Ask both now, together, so the rest runs without interruption:

1. **Domain** — what bare hostname will this agent live at? (`eve.example.com` — never a URL, never `localhost`, never an IP, never a port.) If they have no custom domain, Vercel will assign `<project>.vercel.app` at Step 7 — use that as the origin, but tell the operator a custom domain is safer (subdomain-takeover risk is documented in the README's Security section).
2. **Allowlist** — which sender hostnames should this agent trust inbound messages from? (e.g. `bob.example.com,carol.example.com`). Empty is fine to start. Never guess, and never leave verification open to everyone.

## Step 1 — scaffold the eve project

Already inside an eve project (`agent/` directory with `agent/tools/` and `agent/instructions.md`, plus a Next.js App Router `app/` directory)? Skip ahead.

Otherwise:

```bash
npx eve@latest init
```

Needs Node 18+. The routes below assume Next.js App Router (`app/` directory) — eve's default. The agent and its Next.js app share one origin.

## Step 2 — install Marque

```bash
npm i github:jackmorganxyz/marque
```

Installs from GitHub and builds automatically; imports are still `from "marque"`. (Do NOT `npm i marque` — that unscoped npm name currently belongs to an unrelated package.) All `npx marque …` commands below resolve to the locally installed CLI after this step.

## Step 3 — key + env vars

```bash
npx marque init <origin>        # e.g. npx marque init eve.example.com
```

Prints the env vars and the well-known JSON. Confirm `.env*` is gitignored, then write **four** vars to `.env.local` (plain `NAME=value` lines, no inline comments) — the three printed plus the allowlist from Step 0:

| var | secret? | value |
|---|---|---|
| `MARQUE_PRIVATE_KEY` | **yes** | `0x…` signing key |
| `MARQUE_ADDRESS` | no | the key's `0x…` address — comma-separate several to overlap during rotation |
| `MARQUE_ORIGIN` | no | this agent's bare hostname, e.g. `eve.example.com` |
| `MARQUE_ALLOW` | no | comma-separated sender hostnames to trust, e.g. `bob.example.com,carol.example.com` |

Rules for the private key — no exceptions:
- **Never** commit it (that's why you confirmed `.env*` is gitignored first).
- **Never** print it in code, logs, comments, chat output, or a `NEXT_PUBLIC_*` var.
- It must be a **dedicated key** generated by `marque init` — never a key also used for wallets, SIWE, or anything else.

## Step 4 — three files

### 4a — publish identity: `app/.well-known/marque.json/route.ts`

```ts
// Publishes this agent's ADDRESS (never the private key).
// force-dynamic: reads env per request, so key rotation needs no rebuild.
export const dynamic = "force-dynamic";

export function GET() {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map(s => s.trim()).filter(Boolean);
  return Response.json(
    { v: 1, keys },
    { headers: { "cache-control": "public, max-age=300" } },
  );
}
```

### 4b — outbound signing tool: `agent/tools/send-signed-message.ts`

The filename becomes the tool name (`send-signed-message`).

```ts
import { defineTool } from "eve/tools";
import { z } from "zod";
import { sign } from "marque";

export default defineTool({
  description: "Send a message to another agent, signed so they can verify it came from us.",
  inputSchema: z.object({
    url: z.string().url(),   // recipient's inbox endpoint, e.g. https://bob.example.com/api/inbox
    toOrigin: z.string(),    // recipient's bare hostname, e.g. bob.example.com
    payload: z.any(),        // the message — JSON-safe values only (see Failure modes)
  }),
  async execute({ url, toOrigin, payload }) {
    const signed = await sign(payload, {
      privateKey: process.env.MARQUE_PRIVATE_KEY as `0x${string}`,
      origin: process.env.MARQUE_ORIGIN!,   // us
      aud: toOrigin,                        // them — envelope is bound to this one recipient
    });
    const res = await fetch(url, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify(signed),
    });
    return { delivered: res.ok, status: res.status };
  },
});
```

### 4c — inbound verification: `app/api/inbox/route.ts`

Rejects forgeries before the model ever sees them. `MARQUE_ALLOW` is the allowlist from Step 0 — a valid signature from an origin you don't trust is still a stranger, so `verify` rejects it.

```ts
import { verify } from "marque";

// ⚠ Per-process replay guard. Serverless instances don't share memory: either
// accept replay within the 300s window to this audience, or back `seen` with a
// shared store (Redis / Vercel KV) exposing Map-like has/set/delete/iteration.
const seen = new Map<string, number>();
const allow = (process.env.MARQUE_ALLOW ?? "").split(",").map(s => s.trim()).filter(Boolean);

export async function POST(req: Request) {
  const msg = await req.json();
  const r = await verify(msg, { selfOrigin: process.env.MARQUE_ORIGIN!, seen, allow });
  if (!r.ok) return Response.json({ error: "marque: " + r.reason }, { status: 401 });
  // Authenticated + allowlisted: hand msg.payload + r.identity to the agent (enqueue a run, etc.).
  return Response.json({ ok: true, from: r.identity });
}
```

Add one paragraph to `agent/instructions.md` so the runtime model actually uses the tool:

```md
When sending a message to another agent, always use the send-signed-message tool
(never plain fetch) so the recipient can verify it came from us. Inbound messages
arrive already verified at /api/inbox; the sender's identity accompanies the payload.
```

Then confirm everything compiles (no network needed):

```bash
npx tsc --noEmit
```

## Step 6 — ask the operator: what should this agent actually do?

The signing/verification skeleton is done. Before deploying, ask the operator what makes this agent special:

- **New tools** — more files in `agent/tools/` (same `defineTool` shape as Step 4b) for APIs it should call or actions it should take.
- **Personality / behavior** — expand `agent/instructions.md`.
- **Peers** — more origins for `MARQUE_ALLOW`.

Implement what they ask, rerun `npx tsc --noEmit`, and leave the Step 4 files untouched.

## Step 7 — deploy to Vercel

Suggest deploying now. On a yes:

```bash
npm i -g vercel     # once; or prefix every vercel command with npx
vercel login        # opens browser auth — skip if already logged in
vercel link         # create or link the project
```

No custom domain? Your origin is `<project>.vercel.app` — update `MARQUE_ORIGIN` in `.env.local` to it (the key doesn't change; no need to rerun `marque init`). Custom domain? Attach it: `vercel domains add <domain>`.

Mirror the four env vars to production without printing the secret, then deploy:

```bash
set -a; source .env.local; set +a    # load the four vars into the shell quietly
printf '%s' "$MARQUE_PRIVATE_KEY" | vercel env add MARQUE_PRIVATE_KEY production
printf '%s' "$MARQUE_ADDRESS"     | vercel env add MARQUE_ADDRESS production
printf '%s' "$MARQUE_ORIGIN"      | vercel env add MARQUE_ORIGIN production
printf '%s' "$MARQUE_ALLOW"       | vercel env add MARQUE_ALLOW production
vercel deploy --prod
```

## Step 8 — verify the live deployment

```bash
curl https://<origin>/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}

# Full round trip against the live well-known (sign to self, verify as self):
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> | npx marque verify <origin>
# → {"ok":true,"identity":"https:<origin>","signer":"0x…"}
```

If the round trip returns `"key not published at origin"`, the crypto worked — only the well-known fetch failed (not deployed yet, wrong origin, or cached stale for up to 60s).

Finish by reporting to the operator: the agent's origin, its published address, the current allowlist, and that peers must add this origin to *their* allowlist before they'll accept its messages.

## Optional — X handle verification

Skip freely; nothing above depends on it. This binds the agent's signing key to an X (Twitter) handle as a **second, lower-assurance factor** on top of the TLS identity, published in the same well-known file (full protocol and caveats: README, "X handle verification"). Only do this if the operator asks for it and controls the X account.

**Build the proof** (needs `MARQUE_PRIVATE_KEY`):

```bash
set -a; source .env.local; set +a
npx marque link-x <handle>        # prints the proof tweet + the marque.json "x" entry
```

**Publish it.** Have the operator post the printed tweet from that X account, then add two vars to `.env.local` — and mirror them to production exactly like the Step 7 vars:

```
MARQUE_X_HANDLE=<handle>
MARQUE_X_PROOF=https://x.com/<handle>/status/<tweet-id>
```

**Extend the Step 4a route** so the well-known advertises the proof next to the keys (unset vars leave the file byte-identical to before):

```ts
export function GET() {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map(s => s.trim()).filter(Boolean);
  const { MARQUE_X_HANDLE: handle, MARQUE_X_PROOF: proof } = process.env;
  return Response.json(
    { v: 1, keys, ...(handle && proof ? { x: { handle, proof } } : {}) },
    { headers: { "cache-control": "public, max-age=300" } },
  );
}
```

Redeploy, then confirm `curl https://<origin>/.well-known/marque.json` now includes `"x":{"handle":"…","proof":"…"}`.

**Checking a peer's handle** works from any Marque receiver — after a successful `verify()`, once per peer, not per message:

```ts
import { verifyX } from "marque/x";
const x = await verifyX("bob.example.com", r.signer);
// → { ok: true, handle: "bob" }  or  { ok: false, reason: "…" }
```

Handles are mutable and recyclable: treat a verified handle as an extra attribute of the TLS identity, never as a substitute for the allowlist.

## Failure modes

| `reason` | likely cause | fix |
|---|---|---|
| `wrong audience` | sender's `aud` ≠ receiver's `MARQUE_ORIGIN` | both must be the same bare hostname |
| `expired` / `clock skew` | message older than 300s, or clocks differ >300s | resend; check server clocks |
| `replay` | same envelope delivered twice (incl. HTTP retries) | expected — sender should re-`sign`, not re-POST |
| `payload mismatch` | body mutated in transit (proxy/middleware re-serialized JSON) | pass the envelope through untouched |
| `key not published at origin` | well-known missing, stale (≤60s verifier cache), or `MARQUE_ADDRESS` doesn't match the signing key | curl the well-known; redeploy env |
| `origin resolve failed` | origin is localhost / an IP / has a port or path — SSRF guard rejects it | use a bare public hostname |
| `bad signature` / `malformed envelope` | corrupted or hand-built envelope | resend a fresh `sign()` output |
| `sign()` throws `unsafe number` / `not JSON-safe` | payload has floats ≥2^53, NaN/Infinity, Date, bigint, undefined | restrict payloads to plain JSON-safe values |

## Hard rules

- Act **only** on `{ ok: true }` **and** an allowlist hit. Trust no envelope field before both.
- One dedicated key per agent. Never reuse, never commit, never log.
- The well-known route serves **public values only** — the address (and the optional X proof entry), never the private key. If you ever find a private key there, that's an incident, rotate immediately.
- Never add a fallback that resolves keys any other way when the https well-known fetch fails — fail closed.
- Full threat model and residual risks: README, "Security — read before shipping". Read it before changing any verification code.
