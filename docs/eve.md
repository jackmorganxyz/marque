# Integrating Marque into a Vercel eve agent

> **Audience: AI coding agents.** This file is self-contained — paste it (or point your agent at it) and it has everything needed to add Marque signing + verification to an existing [eve](https://vercel.com/eve) project. Humans welcome too.

Marque lets this agent **sign** outbound messages with its own secp256k1 key and **verify** which https origin sent any inbound message — no shared secret, no registry, no chain. Identity is anchored at `https://<origin>/.well-known/marque.json`, served from the agent's own domain over TLS.

You will create **three files**, set **three env vars**, and add **one line of agent instructions**. Marque's core does not depend on eve; this is the whole adapter.

## Preconditions — check before writing code

1. An eve project: `agent/` directory with `agent/tools/` and `agent/instructions.md` exists (created by `npx eve@latest init`). The agent and its Next.js app share one origin.
2. Next.js App Router (`app/` directory) — the well-known and inbox routes below assume it.
3. Node 18+.
4. The project deploys to a **real public https domain** the operator controls. Marque origins are bare hostnames — `eve.example.com`, never `https://eve.example.com`, never `localhost`, never an IP, never a port. If you only have a `*.vercel.app` preview URL, it works, but flag to the operator that a custom domain is safer (subdomain-takeover risk is documented in the README's Security section).

If any precondition fails, stop and report it instead of improvising.

## Step 1 — install

```bash
npm i github:jackmorganxyz/marque
```

Installs from GitHub and builds automatically; imports are still `from "marque"`. (Do NOT `npm i marque` — that unscoped npm name currently belongs to an unrelated package.) All `npx marque …` commands below resolve to the locally installed CLI after this step.

## Step 2 — key + env vars

```bash
npx marque init <origin>        # e.g. npx marque init eve.example.com
```

Prints the three env vars and the well-known JSON. Put them in `.env.local`, and mirror to production with `vercel env add <NAME> production`.

| var | secret? | value |
|---|---|---|
| `MARQUE_PRIVATE_KEY` | **yes** | `0x…` signing key |
| `MARQUE_ADDRESS` | no | the key's `0x…` address — comma-separate several to overlap during rotation |
| `MARQUE_ORIGIN` | no | this agent's bare hostname, e.g. `eve.example.com` |

Rules for the private key — no exceptions:
- **Never** commit it (confirm `.env*` is gitignored before writing the file).
- **Never** print it in code, logs, comments, chat output, or a `NEXT_PUBLIC_*` var.
- It must be a **dedicated key** generated by `marque init` — never a key also used for wallets, SIWE, or anything else.

## Step 3 — publish identity: `app/.well-known/marque.json/route.ts`

```ts
// Publishes this agent's ADDRESS (never the private key).
// force-dynamic: reads env per request, so key rotation needs no rebuild.
export const dynamic = "force-dynamic";

export function GET() {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map(s => s.trim()).filter(Boolean);
  return Response.json(
    { v: 1, keys },
    { headers: { "cache-control": "public, max-age=300" } },
  );
}
```

## Step 4 — outbound signing tool: `agent/tools/send-signed-message.ts`

The filename becomes the tool name (`send-signed-message`).

```ts
import { defineTool } from "eve/tools";
import { z } from "zod";
import { sign } from "marque";

export default defineTool({
  description: "Send a message to another agent, signed so they can verify it came from us.",
  inputSchema: z.object({
    url: z.string().url(),   // recipient's inbox endpoint, e.g. https://bob.example.com/api/inbox
    toOrigin: z.string(),    // recipient's bare hostname, e.g. bob.example.com
    payload: z.any(),        // the message — JSON-safe values only (see Failure modes)
  }),
  async execute({ url, toOrigin, payload }) {
    const signed = await sign(payload, {
      privateKey: process.env.MARQUE_PRIVATE_KEY as `0x${string}`,
      origin: process.env.MARQUE_ORIGIN!,   // us
      aud: toOrigin,                        // them — envelope is bound to this one recipient
    });
    const res = await fetch(url, {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify(signed),
    });
    return { delivered: res.ok, status: res.status };
  },
});
```

## Step 5 — inbound verification: `app/api/inbox/route.ts`

Rejects forgeries before the model ever sees them. `MARQUE_ALLOW` is a comma-separated allowlist of sender hostnames (e.g. `bob.example.com,carol.example.com`) — a valid signature from an origin you don't trust is still a stranger, so `verify` rejects it.

```ts
import { verify } from "marque";

// ⚠ Per-process replay guard. Serverless instances don't share memory: either
// accept replay within the 300s window to this audience, or back `seen` with a
// shared store (Redis / Vercel KV) exposing Map-like has/set/delete/iteration.
const seen = new Map<string, number>();
const allow = (process.env.MARQUE_ALLOW ?? "").split(",").map(s => s.trim()).filter(Boolean);

export async function POST(req: Request) {
  const msg = await req.json();
  const r = await verify(msg, { selfOrigin: process.env.MARQUE_ORIGIN!, seen, allow });
  if (!r.ok) return Response.json({ error: "marque: " + r.reason }, { status: 401 });
  // Authenticated + allowlisted: hand msg.payload + r.identity to the agent (enqueue a run, etc.).
  return Response.json({ ok: true, from: r.identity });
}
```

Add `MARQUE_ALLOW` to `.env.local` + Vercel env. Ask the operator which origins to trust — do not guess or leave it open.

## Step 6 — tell the agent when to sign: `agent/instructions.md`

Add one paragraph so the runtime model actually uses the tool:

```md
When sending a message to another agent, always use the send-signed-message tool
(never plain fetch) so the recipient can verify it came from us. Inbound messages
arrive already verified at /api/inbox; the sender's identity accompanies the payload.
```

## Verify the integration

Before deploy (no network needed):

```bash
npx tsc --noEmit       # the three new files typecheck
```

After deploy:

```bash
curl https://<origin>/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}

# Full round trip against the live well-known (sign to self, verify as self):
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> | npx marque verify <origin>
# → {"ok":true,"identity":"https:<origin>","signer":"0x…"}
```

If the round trip returns `"key not published at origin"`, the crypto worked — only the well-known fetch failed (not deployed yet, wrong origin, or cached stale for up to 60s).

## Failure modes

| `reason` | likely cause | fix |
|---|---|---|
| `wrong audience` | sender's `aud` ≠ receiver's `MARQUE_ORIGIN` | both must be the same bare hostname |
| `expired` / `clock skew` | message older than 300s, or clocks differ >300s | resend; check server clocks |
| `replay` | same envelope delivered twice (incl. HTTP retries) | expected — sender should re-`sign`, not re-POST |
| `payload mismatch` | body mutated in transit (proxy/middleware re-serialized JSON) | pass the envelope through untouched |
| `key not published at origin` | well-known missing, stale (≤60s verifier cache), or `MARQUE_ADDRESS` doesn't match the signing key | curl the well-known; redeploy env |
| `origin resolve failed` | origin is localhost / an IP / has a port or path — SSRF guard rejects it | use a bare public hostname |
| `bad signature` / `malformed envelope` | corrupted or hand-built envelope | resend a fresh `sign()` output |
| `sign()` throws `unsafe number` / `not JSON-safe` | payload has floats ≥2^53, NaN/Infinity, Date, bigint, undefined | restrict payloads to plain JSON-safe values |

## Hard rules

- Act **only** on `{ ok: true }` **and** an allowlist hit. Trust no envelope field before both.
- One dedicated key per agent. Never reuse, never commit, never log.
- The well-known route serves the **address only** — if you ever find a private key there, that's an incident, rotate immediately.
- Never add a fallback that resolves keys any other way when the https well-known fetch fails — fail closed.
- Full threat model and residual risks: README, "Security — read before shipping". Read it before changing any verification code.
