# Marque + OpenClaw quick start — zero to a publicly verifiable signing agent

> **Audience: AI coding agents.** This file is self-contained — paste it (or point your agent at it) and it walks you from an existing [OpenClaw](https://openclaw.ai/) install to a **deployed** Gateway plugin that signs every outbound agent-to-agent message and verifies every inbound one against an allowlist, published on a public domain. Humans welcome too.

Marque lets this agent **sign** outbound messages with its own secp256k1 key and **verify** which https origin sent any inbound message — no shared secret, no registry, no chain. Identity is anchored at `https://<origin>/.well-known/marque.json`, served from the agent's own domain over TLS.

OpenClaw's Gateway is a single long-running Node process on the operator's machine, and plugins run inside it: one plugin file registers the signing **tool**, the **well-known** route, and the verified **inbox** route. The Gateway binds to loopback, so the last step puts those two routes — and only those two — behind a public https domain.

The path: **two questions for the operator → scaffold the plugin → install Marque → keys → one file → install + restart → customize → publish the two routes → verify live.** Follow the steps in order; if one fails, stop and report it instead of improvising.

## Step 0 — two questions for the operator

Ask both now, together, so the rest runs without interruption:

1. **Domain** — what bare public hostname will this agent live at? (`eve.example.com` — never a URL, never `localhost`, never an IP, never a port.) It must resolve to a machine that can reach the Gateway host (Step 7 puts a reverse proxy there). No domain at all? A [Tailscale Funnel](https://tailscale.com/kb/1223/funnel) hostname (`<machine>.<tailnet>.ts.net`) works — use that as the origin, but tell the operator a domain they own is safer.
2. **Allowlist** — which sender hostnames should this agent trust inbound messages from? (e.g. `bob.example.com,carol.example.com`). Empty is fine to start — the inbox stays closed until it's filled. Never guess, and never leave verification open to everyone.

## Step 1 — scaffold the plugin package

This guide assumes a working OpenClaw install (Gateway 2026.6.x or newer, Node 22.19+) — `openclaw --version` to confirm. In a new directory next to (not inside) the OpenClaw workspace:

```bash
mkdir openclaw-marque && cd openclaw-marque
```

`package.json`:

```json
{
  "name": "openclaw-marque",
  "version": "1.0.0",
  "type": "module",
  "openclaw": {
    "extensions": ["./index.ts"],
    "compat": {
      "pluginApi": ">=2026.6.11",
      "minGatewayVersion": "2026.6.11"
    }
  }
}
```

`openclaw.plugin.json` (every plugin needs a manifest; the tool must be declared in `contracts.tools`):

```json
{
  "id": "marque",
  "name": "Marque",
  "description": "Sign outbound agent-to-agent messages and verify inbound ones with Marque.",
  "contracts": {
    "tools": ["send_signed_message"]
  },
  "activation": {
    "onStartup": true
  },
  "configSchema": {
    "type": "object",
    "additionalProperties": false
  }
}
```

## Step 2 — install Marque (and the plugin's one other dep)

```bash
npm i github:jackmorganxyz/marque typebox
```

Marque installs from GitHub and builds automatically; imports are still `from "marque"`. (Do NOT `npm i marque` — that unscoped npm name currently belongs to an unrelated package.) `typebox` is OpenClaw's tool-parameter schema library. All `npx marque …` commands below resolve to the locally installed CLI after this step.

## Step 3 — key + env vars

```bash
npx marque init <origin>        # e.g. npx marque init eve.example.com
```

Prints the env vars and the well-known JSON. The plugin runs **inside the Gateway process**, so set **four** vars in the environment that launches the Gateway (shell profile, systemd unit, launchd plist — wherever the operator starts `openclaw`): the three printed plus the allowlist from Step 0.

| var | secret? | value |
|---|---|---|
| `MARQUE_PRIVATE_KEY` | **yes** | `0x…` signing key |
| `MARQUE_ADDRESS` | no | the key's `0x…` address — comma-separate several to overlap during rotation |
| `MARQUE_ORIGIN` | no | this agent's bare public hostname, e.g. `eve.example.com` |
| `MARQUE_ALLOW` | no | comma-separated sender hostnames to trust, e.g. `bob.example.com,carol.example.com` |

Rules for the private key — no exceptions:
- **Never** commit it, and never put it in `openclaw.json` (that file is often synced/backed up).
- **Never** print it in code, logs, comments, or chat output.
- It must be a **dedicated key** generated by `marque init` — never a key also used for wallets, SIWE, or anything else.

## Step 4 — one file

`index.ts` — the whole integration: the signing tool, the well-known route, and the verified inbox. Inbound messages that pass `verify()` are enqueued as a system event for the operator's main session (`agent:main:main`) and the Gateway is woken — the model only ever sees authenticated, allowlisted messages. Both routes use `auth: "plugin"` deliberately: the well-known is public by design, and on the inbox `verify()` **is** the authentication.

```ts
import { Type } from "typebox";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { sign, verify } from "marque";

// Replay guard. The Gateway is one long-lived process, so unlike serverless
// hosts this Map genuinely bounds replay for the whole deployment.
const seen = new Map<string, number>();

const SESSION_KEY = "agent:main:main";   // operator's main session receives verified messages
const MAX_BODY = 64 * 1024;

async function readJsonBody(req: AsyncIterable<unknown>): Promise<unknown> {
  const chunks: Buffer[] = [];
  let size = 0;
  for await (const chunk of req) {
    const buf = chunk as Buffer;
    size += buf.length;
    if (size > MAX_BODY) throw new Error("body too large");
    chunks.push(buf);
  }
  return JSON.parse(Buffer.concat(chunks).toString("utf8"));
}

export default definePluginEntry({
  id: "marque",
  name: "Marque",
  description: "Sign outbound agent-to-agent messages and verify inbound ones with Marque.",
  register(api) {
    // Outbound: the model calls this instead of plain fetch, so recipients can verify us.
    api.registerTool({
      name: "send_signed_message",
      label: "Send signed message",
      description:
        "Send a message to another agent, signed with this agent's Marque key so the " +
        "recipient can verify it came from us. Use this instead of fetch for agent-to-agent messages.",
      parameters: Type.Object({
        url: Type.String({ description: "recipient inbox endpoint, e.g. https://bob.example.com/marque/inbox" }),
        toOrigin: Type.String({ description: "recipient bare hostname, e.g. bob.example.com" }),
        payload: Type.Any({ description: "the message — JSON-safe values only" }),
      }),
      async execute(_id, params) {
        const { url, toOrigin, payload } = params as { url: string; toOrigin: string; payload: unknown };
        const signed = await sign(payload, {
          privateKey: process.env.MARQUE_PRIVATE_KEY as `0x${string}`,
          origin: process.env.MARQUE_ORIGIN!,   // us
          aud: toOrigin,                        // them — envelope is bound to this one recipient
        });
        const res = await fetch(url, {
          method: "POST",
          headers: { "content-type": "application/json" },
          body: JSON.stringify(signed),
        });
        const outcome = { delivered: res.ok, status: res.status };
        return { content: [{ type: "text", text: JSON.stringify(outcome) }], details: outcome };
      },
    });

    // Publishes this agent's ADDRESS (never the private key).
    api.registerHttpRoute({
      path: "/.well-known/marque.json",
      auth: "plugin",
      handler: (_req, res) => {
        const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map((s) => s.trim()).filter(Boolean);
        res.statusCode = 200;
        res.setHeader("content-type", "application/json");
        res.setHeader("cache-control", "public, max-age=300");
        res.end(JSON.stringify({ v: 1, keys }));
        return true;
      },
    });

    // Inbox: rejects forgeries before the model ever sees them. MARQUE_ALLOW is the
    // allowlist from Step 0 — a valid signature from an origin you don't trust is
    // still a stranger, so verify rejects it.
    api.registerHttpRoute({
      path: "/marque/inbox",
      auth: "plugin",
      handler: async (req, res) => {
        const respond = (status: number, body: unknown) => {
          res.statusCode = status;
          res.setHeader("content-type", "application/json");
          res.end(JSON.stringify(body));
          return true;
        };
        if (req.method !== "POST") return respond(405, { error: "POST only" });
        let msg: unknown;
        try {
          msg = await readJsonBody(req);
        } catch {
          return respond(400, { error: "invalid JSON body" });
        }
        const allow = (process.env.MARQUE_ALLOW ?? "").split(",").map((s) => s.trim()).filter(Boolean);
        const r = await verify(msg as never, { selfOrigin: process.env.MARQUE_ORIGIN!, seen, allow });
        if (!r.ok) return respond(401, { error: "marque: " + r.reason });
        // Authenticated + allowlisted: deliver payload + proven identity to the agent.
        api.runtime.system.enqueueSystemEvent(
          `Verified Marque message from ${r.identity}: ${JSON.stringify((msg as { payload: unknown }).payload)}`,
          { sessionKey: SESSION_KEY },
        );
        api.runtime.system.requestHeartbeat({ source: "other", intent: "event", reason: "marque-inbound" });
        return respond(202, { ok: true, from: r.identity });
      },
    });
  },
});
```

## Step 5 — install the plugin, restart, smoke-test locally

```bash
openclaw plugins install --link ./openclaw-marque
openclaw gateway restart
openclaw plugins inspect marque --runtime
```

`inspect` should show the loaded runtime with the tool and both HTTP routes. Then hit the Gateway directly (default loopback port 18789):

```bash
curl http://127.0.0.1:18789/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}
```

An empty `keys` array means the Gateway process can't see `MARQUE_ADDRESS` — fix Step 3's env placement and restart. Don't POST to the local inbox yet: `verify` fetches the sender's key from the **live** `https://<origin>/.well-known/marque.json`, so inbound verification only round-trips after Step 7.

## Step 6 — one paragraph of agent instructions, then customize

Add one paragraph to the OpenClaw agent's workspace instructions (`AGENTS.md` in the agent workspace) so the model actually uses the tool:

```md
When sending a message to another agent, always use the send_signed_message tool
(never plain fetch or curl) so the recipient can verify it came from us. Inbound
messages arrive already verified as system events; the sender's proven identity
accompanies the payload.
```

Then ask the operator what else this agent should do — more peers for `MARQUE_ALLOW`, a different receiving session (edit `SESSION_KEY` if messages should land somewhere other than the main session), personality in `AGENTS.md`. Leave the Step 4 signing/verification code untouched.

## Step 7 — publish the two routes on the domain

The Gateway stays on loopback. Put a reverse proxy on the machine the domain points at, forwarding **only** the two Marque paths — never the whole Gateway (dashboard, WebSocket, and RPC surfaces must stay private). With [Caddy](https://caddyserver.com/) (automatic TLS), `/etc/caddy/Caddyfile`:

```
eve.example.com {
    reverse_proxy /.well-known/marque.json 127.0.0.1:18789
    reverse_proxy /marque/inbox 127.0.0.1:18789
    respond 404
}
```

```bash
sudo systemctl reload caddy
```

No domain? Tailscale Funnel can publish the Gateway port with TLS at `<machine>.<tailnet>.ts.net` (see OpenClaw's Tailscale docs) — but Funnel exposes the whole port, so prefer the reverse-proxy path allowlist whenever possible. Whatever the public hostname ends up being, it must equal `MARQUE_ORIGIN` exactly — update the env var and restart the Gateway if it changed.

## Step 8 — verify the live deployment

```bash
curl https://<origin>/.well-known/marque.json
# → {"v":1,"keys":["0x<MARQUE_ADDRESS>"]}

# Full round trip against the live well-known (sign to self, verify as self):
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> | npx marque verify <origin>
# → {"ok":true,"identity":"https:<origin>","signer":"0x…"}
```

If the round trip returns `"key not published at origin"`, the crypto worked — only the well-known fetch failed (proxy not forwarding, wrong origin, or cached stale for up to 60s). To exercise the inbox end-to-end, temporarily add `<origin>` itself to `MARQUE_ALLOW` (restart the Gateway) and POST a self-signed envelope — this wakes the agent with a real message:

```bash
echo '{"ping":1}' | MARQUE_PRIVATE_KEY=0x… npx marque sign <origin> <origin> \
  | curl -sS -X POST https://<origin>/marque/inbox -H 'content-type: application/json' -d @-
# → {"ok":true,"from":"https:<origin>"}
```

Finish by reporting to the operator: the agent's origin, its published address, the current allowlist, and that peers must add this origin to *their* allowlist before they'll accept its messages.

## Optional — X handle verification

Skip freely; nothing above depends on it. This binds the agent's signing key to an X (Twitter) handle as a **second, lower-assurance factor** on top of the TLS identity, published in the same well-known file (full protocol and caveats: README, "X handle verification"). Only do this if the operator asks for it and controls the X account.

**Build the proof** (needs `MARQUE_PRIVATE_KEY` — run in the same environment that holds the Step 3 vars):

```bash
npx marque link-x <handle>        # prints the proof tweet + the marque.json "x" entry
```

**Publish it.** Have the operator post the printed tweet from that X account, then set two more vars where the Gateway is launched (next to the Step 3 vars) and restart the Gateway:

```
MARQUE_X_HANDLE=<handle>
MARQUE_X_PROOF=https://x.com/<handle>/status/<tweet-id>
```

**Extend the Step 4 well-known handler** so it advertises the proof next to the keys (unset vars leave the file byte-identical to before):

```ts
handler: (_req, res) => {
  const keys = (process.env.MARQUE_ADDRESS ?? "").split(",").map((s) => s.trim()).filter(Boolean);
  const { MARQUE_X_HANDLE: handle, MARQUE_X_PROOF: proof } = process.env;
  res.statusCode = 200;
  res.setHeader("content-type", "application/json");
  res.setHeader("cache-control", "public, max-age=300");
  res.end(JSON.stringify({ v: 1, keys, ...(handle && proof ? { x: { handle, proof } } : {}) }));
  return true;
},
```

Restart the Gateway, then confirm `curl https://<origin>/.well-known/marque.json` now includes `"x":{"handle":"…","proof":"…"}`.

**Checking a peer's handle** works from any Marque receiver — after a successful `verify()`, once per peer, not per message:

```ts
import { verifyX } from "marque/x";
const x = await verifyX("bob.example.com", r.signer);
// → { ok: true, handle: "bob" }  or  { ok: false, reason: "…" }
```

Handles are mutable and recyclable: treat a verified handle as an extra attribute of the TLS identity, never as a substitute for the allowlist.

## Failure modes

| `reason` | likely cause | fix |
|---|---|---|
| `wrong audience` | sender's `aud` ≠ receiver's `MARQUE_ORIGIN` | both must be the same bare hostname |
| `expired` / `clock skew` | message older than 300s, or clocks differ >300s | resend; check server clocks |
| `replay` | same envelope delivered twice (incl. HTTP retries) | expected — sender should re-`sign`, not re-POST |
| `payload mismatch` | body mutated in transit (proxy re-serialized JSON) | proxy must pass the body through untouched |
| `key not published at origin` | well-known missing, stale (≤60s verifier cache), or `MARQUE_ADDRESS` doesn't match the signing key | curl the well-known; fix env; restart Gateway |
| `origin resolve failed` | origin is localhost / an IP / has a port or path — SSRF guard rejects it | use a bare public hostname |
| `sender not allowed` | sender's origin missing from `MARQUE_ALLOW` (an empty allowlist rejects everyone) | add the peer's hostname; restart Gateway |
| `bad signature` / `malformed envelope` | corrupted or hand-built envelope | resend a fresh `sign()` output |
| `sign()` throws `unsafe number` / `not JSON-safe` | payload has floats ≥2^53, NaN/Infinity, Date, bigint, undefined | restrict payloads to plain JSON-safe values |

OpenClaw-specific:

| symptom | likely cause | fix |
|---|---|---|
| tool or routes missing after install | Gateway not restarted, or plugin failed to load | `openclaw gateway restart`; `openclaw plugins inspect marque --runtime` |
| `keys: []` from the well-known | env vars not visible to the Gateway process | set them where the Gateway is launched (Step 3); restart |
| tool registered but model never calls it | tool name missing from `contracts.tools`, or instructions not added | check the manifest and Step 6 |
| public URL 404s but loopback works | reverse proxy path mismatch | proxy must forward the exact paths from Step 7 |
| inbound accepted but agent never reacts | wrong `SESSION_KEY`, or heartbeat disabled | confirm the session key; check Gateway heartbeat config |

## Hard rules

- Act **only** on `{ ok: true }` **and** an allowlist hit. Trust no envelope field before both.
- One dedicated key per agent. Never reuse, never commit, never log.
- The well-known route serves **public values only** — the address (and the optional X proof entry), never the private key. If you ever find a private key there, that's an incident, rotate immediately.
- Never add a fallback that resolves keys any other way when the https well-known fetch fails — fail closed.
- Never expose the rest of the Gateway (dashboard, WebSocket, RPC) on the public hostname — only the two Marque paths.
- Full threat model and residual risks: README, "Security — read before shipping". Read it before changing any verification code.
